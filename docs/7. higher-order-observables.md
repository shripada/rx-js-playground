## Higher order observables

An observable that emits other observables is known as a `higher-order observable`.

```typescript
Observable<Observable<number>>;
```

The outer observable emits inner observables. To get the actual values, those inner observables must also be subscribed to.

```typescript
const numberHOObservable$ = of(of(1), of(2), of(3));

numberHOObservable$.subscribe((innerObservable) => {
  innerObservable.subscribe((value) => console.log(value));
});
```

In practice we usually do not manually subscribe twice like this. RxJS provides flattening operators that subscribe to the inner observables for us.

## Creating a higher-order observable with `map`

The file starts with a normal observable such as:

```typescript
const numberObservable$ = of(1, 2, 3, 4, 5);
```

If we transform each number into an observable, the result becomes a higher-order observable.

```typescript
const numberHOObservableUsingPipe$ = numberObservable$.pipe(
  map((value) => of(value)),
);
```

This is very similar to arrays:

```typescript
const numbers = [1, 2, 3, 4];
const transformed = numbers.map((value) => [value]);
const flattened = transformed.flat();
const transformedAndFlattened = numbers.flatMap((value) => [value]);
```

With observables, `map` can create nested observables and a flattening operator can turn them back into a regular stream.

## `tap` and `concatAll`

Here is an example, that demonstrates how we end up with a higher order observable, because of
a map whose transform function returns an observable.

```typescript
numberObservable$
  .pipe(
    tap((value) => {
      console.log(value); // Use tap for side effects
    }),
    map((value) => of(value)),
    concatAll(), // flattens the input observable.
  )
  .subscribe((value) => console.log(value));
```

Important points:

- `tap` is only for side effects such as logging.
- Returning a value from `tap` does not replace the stream value.
- `map((value) => of(value))` creates a higher-order observable.
- `concatAll()` flattens that higher-order observable by subscribing to one inner observable at a time.

## `concatAll`

`concatAll` processes inner observables sequentially.

```typescript
const numberHOObservable1$ = of(
  interval(1000).pipe(take(3)),
  of(2),
  of(3),
  of(4),
  of(5),
);

numberHOObservable1$.pipe(concatAll()).subscribe((value) => console.log(value));
```

The first inner observable must complete before the next one is subscribed to. That is why `concatAll` preserves order.

This is useful when order matters, or when each inner task should start only after the previous one finishes.

## `mergeAll`

`mergeAll` also flattens a higher-order observable, but it subscribes to incoming inner observables immediately instead of waiting for the previous one to complete.

```typescript
numberHOObservable1$.pipe(mergeAll()).subscribe((value) => console.log(value));
```

Because the inner observables run concurrently, values are emitted as soon as they arrive. Order is not guaranteed.

Use `mergeAll` when concurrency is desired and strict ordering does not matter.

## `concatMap` and `mergeMap`

Writing `map(... => observable)` and then `concatAll()` or `mergeAll()` is common enough that RxJS provides shorthand operators.

```typescript
concatMap(transform) === map(transform) + concatAll();
mergeMap(transform) === map(transform) + mergeAll();
```

Example usages:

```typescript
const numberObservable1$ = from([10, 11, 12, 13]);

numberObservable1$
  .pipe(concatMap((value) => of(value)))
  .subscribe((value) => console.log(value));

numberObservable1$
  .pipe(mergeMap((value) => of(value)))
  .subscribe((value) => console.log(value));
```

Key difference:

- `concatMap` keeps the input order by waiting for each inner observable to complete.
- `mergeMap` starts inner observables immediately and merges their outputs as they arrive.

## `switchMap`

`switchMap` is different from both `concatMap` and `mergeMap`.

For each outer value, it subscribes to a new inner observable and unsubscribes from the previous inner observable.

```typescript
obs2
  .pipe(
    switchMap((value) =>
      interval(value).pipe(
        take(1),
        map(() => {
          switch (value) {
            case 1000:
              return 'one thousand';
            case 2000:
              return 'two thousand';
            case 3000:
              return 'three thousand';
            case 4000:
              return 'four thousand';
          }
        }),
      ),
    ),
  )
  .subscribe((value) => console.log(value));
```

This means `switchMap` always prefers the latest inner observable.

It is useful when older work should be discarded, such as autocomplete requests or reacting only to the latest user action.

### A subtle `switchMap` detail

The source also includes this example:

```typescript
const obs3 = of(3000, 1000, 4000, 2000);
obs3.pipe(switchMap((value) => of(value))).subscribe((value) => {
  console.log(value);
});
```

Here the inner observable `of(value)` emits immediately. Because the inner observable finishes so quickly, there may be nothing left to cancel by the time the next outer value arrives. In cases like this, `switchMap` can still appear to emit every value.

So the cancellation behavior of `switchMap` becomes most visible when inner observables stay active long enough to be replaced.

## Choosing the right operator

- Use `concatAll` or `concatMap` when order matters and inner observables should run one after another.
- Use `mergeAll` or `mergeMap` when inner observables can run concurrently.
- Use `switchMap` when only the latest inner observable should remain active.

## Reading the final pipeline in the source

The last part of the file moves to a more realistic example where each input value is turned into another observable. That is still the same higher-order idea:

- the outer stream emits user ids
- each id is mapped to an observable that fetches one user
- `concatMap` flattens those inner observables sequentially
- `map((user) => user.name)` transforms each emitted user object into just the user name

That final `map((user) => user.name)` step is what makes the subscriber receive names instead of full user objects.
